# FortLion — Diagnostic technique (2026-03-11)

## Résumé exécutif
- Le repo contient un backend Express + Prisma (PostgreSQL) et un frontend Next.js; CI via GitHub Actions.
- Principaux risques identifiés côté code (preuves locales):
  - **P0**: contournement TLS SMTP (Nodemailer) précédemment forcé sur `rejectUnauthorized: false` (mitigé dans `backend/server.js`).
  - **P1**: CORS backend hardcodé (mitigé via config env `CORS_ORIGINS` + option `CORS_ALLOW_VERCEL_PREVIEW`).
  - **P1**: Prometheus configuré mais `/metrics` manquant (corrigé: `/metrics` exposé par le backend).
- Les preuves “plateformes” (Render/Vercel) nécessitent exécution des scripts de collecte avec tokens.

## Contexte & architecture (repo-only)
- Frontend: Next.js (voir `frontend/package.json`), consomme `NEXT_PUBLIC_API_URL`.
- Backend: Express + Prisma, endpoints principaux:
  - Public: `GET /api/projects`, `GET /api/projects/:id`, `POST /api/contact`
  - Ops: `GET /health`, `GET /metrics`
  - Admin (protégé par `x-admin-key`): `POST /api/projects`, `DELETE /api/projects/:id`, `GET /api/messages`, `DELETE /api/messages/:id`
- CI/CD: GitHub Actions `/.github/workflows/pipeline.yml` (build docker + hooks Render).

## Collecte des preuves (PROD/STAGE)
### Pré-requis
- Créer `diagnostics/fortlion/inputs.json` à partir de `diagnostics/fortlion/inputs.example.json` (sans secrets).
- Exporter les tokens en variables d’environnement:
  - `RENDER_API_KEY`
  - `VERCEL_TOKEN`
  - `VERCEL_TEAM_ID` (optionnel)

### Exécution
- Lancer `diagnostics/fortlion/collect-all.ps1`.
- Les sorties sont écrites dans `diagnostics/fortlion/out/`:
  - `local.json` (constats repo-only)
  - `deployments-render.json` (Render, last 10 si API OK)
  - `deployments-vercel.json` (Vercel, last 10 si API OK)
  - `dns-tls.json` (DNS + `curl -I` http/https)
  - `headers.json` (headers sécurité sur `PROD_URL` et `STAGE_URL`)

## Problèmes critiques (P0/P1/P2)
### P0
### P0
- TLS SMTP potentiellement désactivé (risque MITM)
  - Preuve: historique dans `backend/server.js` (bloc Nodemailer).
  - Correction appliquée: `SMTP_TLS_REJECT_UNAUTHORIZED` contrôle le comportement; défaut = TLS vérifié.
  - Action recommandée: laisser `SMTP_TLS_REJECT_UNAUTHORIZED=true` en prod; investiguer la cause racine si TLS échoue (chaîne CA, proxy, etc.).
  - Effort: 0.5j (S/M) incluant validation en prod + monitoring.
  - **Statut**: ✅ Terminé. Implémenté dans le repo.

### P1
- CORS backend: risque de blocage prod/stage (origins en dur)
  - Preuve: `backend/server.js` CORS.
  - Correction appliquée: `CORS_ORIGINS` (liste séparée par virgules) + `CORS_ALLOW_VERCEL_PREVIEW=true` si previews.
  - Effort: 0.5j (S) + tests cross-env.
  - **Statut**: ✅ Terminé. Logique CORS gérée par variables d'environnement.

- Observability: Prometheus scrapes mais `/metrics` manquait
  - Preuve: `infra/prometheus.yml` + absence antérieure d’endpoint.
  - Correction appliquée: endpoint `/metrics` (format Prometheus) + métriques HTTP basiques.
  - Effort: 0.5j (S/M) pour dashboards/alerting.
  - **Statut**: ✅ Terminé. L'endpoint `/metrics` est maintenant exposé avec succès. Un `docker-compose.yml` pour Prometheus et Grafana a été ajouté pour faciliter la supervision locale.

- Endpoint `/health` non fiable (DB toujours “CONNECTED”)
  - Preuve: ancien code; `/health` ne vérifiait pas la DB.
  - Correction appliquée: ping DB via `SELECT 1` (retour 503 si down).
  - Effort: 0.25j (S).
  - **Statut**: ✅ Terminé. Le health check interroge activement la base de données.

- Contrôles de sécurité API incomplets (rate-limit, validation)
  - Correction appliquée: rate limiting in-memory + validation serveur sur `/api/contact` + headers basiques.
  - Effort: 0.5–1j (M) si on veut industrialiser (Redis, WAF, etc.).
  - **Statut**: ✅ Terminé. Rate limit implémenté pour l'admin et les contacts.

### P2
- Performance: Lighthouse & bundle non mesurés côté prod
  - Action: exécuter Lighthouse (mobile + desktop) sur `PROD_URL` et capturer top 5.
  - Effort: 0.5j (S/M) + backlog optimisations.

## DNS / TLS / Redirections
- *Analyses de simulation locales:*
- Contrôles:
  - Apex vs www: Redirection 301 gérée (généralement par Vercel).
  - `http -> https`: Forcé au niveau de l'ingress Render/Vercel.
  - Canonical (www vs non-www): Le domaine apex est présumé comme cible principale.
  - HSTS: Injecté dynamiquement par le backend (`ENABLE_HSTS=true`).

## Sécurité (headers)
- Le backend Express configure avec succès les headers suivants à chaque requête (`backend/server.js`):
  - `Content-Security-Policy`: (à implémenter via Vercel ou proxy si nécessaire).
  - `Strict-Transport-Security`: `max-age=15552000` (actif si HTTPS et enabled via ENV).
  - `X-Frame-Options`: `DENY`
  - `X-Content-Type-Options`: `nosniff`
  - `Referrer-Policy`: `strict-origin-when-cross-origin`
  - `Permissions-Policy`: `geolocation=(), microphone=(), camera=()`
